# 調査レポート: ツール・ハルシネーションと履歴汚染の根本原因分析

**調査日時**: 2026-03-10
**対象**: OpenClaw Gemini CLI Adapter におけるツール実行のハルシネーション（偽装実行）および記憶喪失問題

## 背景
UI（OpenClaw）上でAIがアクションを実行する際、実際には裏でエラーが起きている（あるいはコマンドが実行されていない）にもかかわらず、画面上は「成功した」とテキストだけで出力して会話を続ける深刻なハルシネーションが発生。

## 調査の道筋（仮説と検証のトレイル）

### 1. 第一の仮説: 「ファイル書き込み自体への失敗」
- **仮説**: 何らかの理由でファイル操作ツールが失敗しているにもかかわらず、アダプタが成功のシグナルを返しているのではないか。
- **検証**: アダプタの動作と、AIが生成したテキストを直接確認。
- **事実の発見**: AIが返してくる生データ（`adapter_last_req.json` の末尾）を確認したところ、そもそもJSONの関数呼び出し（`tool_calls`）を発行しておらず、ただのマークダウンテキストとして `⚙️ Using tool [write_file] ...` という文字列を手書き（生成）しているだけであることが判明。

### 2. 第二の仮説: 「過去の履歴テキストによるフューショット誤学習」
- **仮説**: AIはなぜ「JSONではなく文字列でツール実行を表現する」と思い込んでしまったのか。過去のリクエスト履歴に、これらの文字列がそのまま混入（汚染）しているのではないか。
- **検証**: `streaming.js` の実装を確認し、その出力が履歴にどう影響するかをパースするスクリプト（Node.js）を用いて `adapter_last_req.json` の過去メッセージを調査。
- **事実の発見**: `streaming.js` にて、クライアント（OpenClaw UI）に対し、プログレス表示として `⚙️ Using tool...` や `✅ Tool finished.` というテキストを意図的に挿入・送信していることを確認。クライアントはそれを「AIの発言内容」として保存してしまい、次回の通信時にそのままアダプタへ送り返していた。これにより履歴が装飾文字列で完全に汚染されていた。

### 3. 第三の仮説: 「ネイティブ履歴構造との不一致と、本物のツール結果JSONの消失」
- **仮説**: Gemini CLIは本来ツール実行結果をどう保存しているのか？そして、なぜOpenClawクライアントから戻ってきた履歴には `tool_calls` のJSON構造が残っていないのか？
- **検証**: アダプタを介さず、ネイティブの `gemini -p` コマンドで `date` コマンドを実行させ、`~/.gemini/tmp/.../chats/session-*.json` に保存された純正のセッション履歴の構造を調査。続けて `converter.js` および `session.js` の履歴変換ロジックを精査。
- **事実の発見**:
  1. ネイティブのGemini CLIの履歴構造には、テキストの `content` とは完全に分離された `toolCalls` 配列があり、そこに引数やシェル実行結果の長文Outputが完全にパッキングされている。
  2. アダプタの `session.js` における `overwriteSessionHistory()` は、送信されてきた「テキストのみの不完全な履歴」を使って、**Gemini CLIの完璧なセッションJSONを丸ごと上書きして破壊**していた。

## 結論と根本的メカニズム

今回のハルシネーションは以下のような死の連鎖（デス・スパイラル）によって引き起こされていた。

1. **ターン1**: AIが正しくJSONでツールを呼ぶ。Gemini CLIは `toolCalls` を保存し、アダプタはUIに `⚙️ Using tool...` というテキストプログレスを送信する。
2. **ターン2**: ユーザーが次の質問を送る。UIは「前回AIは `⚙️ Using tool...` と喋った」という履歴データをアダプタに送り返してくる。
3. **履歴の破壊**: `overwriteSessionHistory()` が、その送られてきた不完全な履歴で Gemini CLI のセッションファイルを上書きする。これにより、ターン1で保存されていたはずの正確な `toolCalls` （操作の記憶と結果）が完全に消滅する。
4. **ハルシネーションの発症**: AIは「過去に自分がどうやってツールを実行したか」の正式な記憶（JSON）を奪われ、残された `⚙️ Using tool...` という生テキストだけを見せられる。「なるほど、このシステムではテキストでこれを出力すればツールが完了したことになるんだな」と誤学習し、一切のツール機能を使わず文字だけで劇を描写し始める。

## 今後の修正方針（`implementation_plan.md`に委譲）
この構造的欠陥を直すため、`src/server.js` にて履歴の文字列サニタイズ（浄化）を行い、`src/session.js` の `overwriteSessionHistory()` において「完全上書き」ではなく「既存の `toolCalls` のマージ・保護」を行うアーキテクチャへの改修が初期提案された。

## 【追加の議論と検証ループ2：ステートレス化の棄却と新たな合成戦略】
- **議論/指摘 (2026-03-10)**: ユーザーより、「`--resume` の廃止（ステートレス化）は不可能である」との強い指摘。Gemini CLIの仕様上、過去の数万トークンに及ぶ全履歴を毎回プロンプト文字列として引数で渡し直すことは非現実的であり、ネイティブのセッションファイル（状態）に頼る仕組みはどうしても捨てられない。
- **新仮説（サバイバル・マージ戦略）**:
  1. クライアントからの履歴（圧縮や配列のズレがある）でGeminiセッションの `messages` を上書きすることは避けられない。
  2. しかし、AIがツール呼び出しに使った「記憶（`toolCalls` オブジェクト）」は絶対に失ってはならない。
  3. UIクライアントのテキスト履歴には、アダプタが埋め込んだ `⚙️ Using tool [tool_name] ...` という**「一意の手がかり（テキストシグネチャ）」**が残っている。
  4. したがって、アダプタの `session.js` の中で**「クライアントのテキスト履歴から `⚙️ Using tool [tool_name]` の文字列パターンを正規表現で走査し、それに一致する `toolCalls` を過去のGeminiマスターセッションファイルから探し出してきて、新しい履歴オブジェクトに移植（レスキュー）する」**という動的な合成手法をとれば、配列インデックスのズレや圧縮にも耐性を持ちつつ、ハルシネーションの発症を防ぐことができるのではないか。
## 【追加の議論と検証ループ3：テキスト汚染の根本的停止】
- **議論/指摘 (2026-03-10)**: ユーザーより、「そもそもアダプタがUIに対して勝手に『⚙️ Using tool...』といったテキストを送信する（テキスト汚染）のをやめれば済む話ではないのか？」という本質的かつ鋭い指摘。
- **検証と事実の再確認**: 
  1. 確かに `src/streaming.js` からその送信処理を消せば、UIのテキストは汚染されず、送信される履歴もクリーンなものになり、AIが「テキストでツール結果を偽装するハルシネーション」は即座に直る。
  2. しかし、それだけでは**「ツールの記憶喪失（Amnesia）」の根本問題が解決しない**という事実が浮上した。
  3. テキスト汚染をなくしても、OpenClawクライアントは「過去に実行された純正のツールコール（JSON）データ」を持っていないため、返送してくる履歴には依然として `toolCalls` が存在しない。
  4. したがって、アダプタが `overwriteSessionHistory()` を無条件で実行すると、せっかくGemini CLI本体が持っていた「正確なツール実行記録（`toolCalls` と長文の実行結果）」が、クライアントからの「綺麗なテキストだけ」の履歴によって**やはり丸ごと上書き・消去されてしまう**。
- **新仮説（結合的アプローチ）**: ハルシネーションと記憶喪失の両方を完全に断ち切るには、以下の二軸での修正が必須である。
  1. ユーザーの指摘通り、アダプタからの「勝手なテキスト汚染（`⚙️ Using tool...` など）」を**完全に停止**し、UIには最終回答のテキストのみを返すようにする（一時的なUX低下よりもシステムの安定を最優先する）。
  2. その上で、アダプタが安易に `overwriteSessionHistory()`（完全上書き）を行うのをやめ、**「ユーザーが明示的に履歴を編集・削除（またはコンテキスト圧縮）した場合のみ、Geminiセッションの長さを調整する」が、「過去のメッセージが持つ `toolCalls` は絶対にGemini本体のJSONから消さない・上書きしない」**という堅牢な状態同期ロジックを実装する。

## 【追加の議論と検証ループ4：真の震源地の特定（テキストへの完全ダウングレード）】
- **議論/指摘 (2026-03-10)**: ユーザーより、「そもそもアダプタがUIクライアント（OpenClaw）へ返答を返す段階で、ツール履歴を削除して単なるテキストにマージして送信しているだけなのでは？だからクライアントにはテキストしか残っていないのではないか」という決定的な指摘。
- **検証と事実の再確認**: `src/streaming.js` におけるSSE送信処理（`case 'tool_use'` および `case 'tool_result'`）をソースコードレベルで精査。
- **事実の発見**: 全くもって指摘の通りであった。アダプタは、Gemini API から送られてきた構造化されたツールデータ（JSON）をOpenClawクライアントに正しく送り返していなかった。
  1. OpenAIの仕様（OpenClawが期待する形式）では、ツールコールの結果は `delta: { tool_calls: [{...}] }` などのJSONオブジェクトとしてストリーミングで一貫して返されなければならない。
  2. しかし現在のアダプタは、それをすべて無視し、`delta: { content: "\n\n⚙️ Using tool..." }` という**単なる文字列（`content`）のチャンクにダウングレードして無理やりストリーミング送信**していた。
  3. このため、OpenClawクライアントの内部・履歴には当然としてJSON構造（`tool_calls`）は一切構築されず、「AIが文字としてそう発言した」というテキストデータしか残らない。
  4. その結果、次回の会話でクライアントが過去のテキストしか返信してこないのはOpenClawとして当然の正常な挙動であり、それをアダプタ（`session.js` の `overwriteSessionHistory`）が真に受けてGeminiセッションを上書きするため、完全な記憶喪失（ツール結果の消失）が物理的かつ必然的に引き起こされていた。

**【最終結論と完全改修方針の見直し】**
アダプタの処理の「入口（サニタイズして復元）」でごまかすのではなく、「出口（レスポンス生成）」を規格通りに正しく直すことが根本解決であると判断された。

## 【追加の議論と検証ループ5：OpenAIストリーミング仕様の構造的限界】
- **議論/指摘 (2026-03-10)**: ユーザーより、「両者が認識するJSON構造をコードを解析してしっかりと洗い出し、相互変換する必要がある」とのさらなる深堀り要求。
- **検証と事実の再確認**: Gemini CLIがローカルに保存する `session-*.json` (ネイティブフォーマット) と、アダプタが変換する先の OpenAI API フォーマット (`tool_calls` と `tool_call_id`) の構造を詳細にマッピング検証した。
- **事実の発見（致命的な不一致）**: 
  1. **Gemini CLI (自律エージェント)**: 1つのユーザープロンプトに対し、内部で（APIを往復せずに）自律的に「思考→ツール実行→結果取得→再思考→最終回答」の多段プロセスを回し、それらをすべて配列として1つの `gemini` メッセージ（`toolCalls` 内に結果も含む）にパックして保持している。
  2. **OpenAI API (1ターン完結型)**: 1回のストリーミングレスポンスにおいて返却できる機能は、「一人のAssistantのテキスト」または「一人のAssistantのツール要求（`tool_calls`）」の**どちらか一つ**だけである。
  3. **構造的限界**: したがって、いくらアダプタが `src/streaming.js` の出口を正しくOpenAI規格（`delta: { tool_calls: [...] }`）に直したところで、もしOpenClawに対し「ツールを実行しろ」というシグナルを送ってしまえば、OpenClaw側（UI）が自前でツールを実行しようとフリーズしてしまう。Gemini CLIは「裏で勝手にツールを実行し終わっている」ため、この「自律的に行われた複数のツール実行履歴と結果」を、OpenAIのストリーミング規格にねじ込んでクライアントに送り返すことは**仕様上 100% 不可能**であることが判明した。

**【真の最終結論（絶対記憶保護・ステートフルマージ）】**
OpenAIのAPI仕様上、サーバー側で勝手に暴れまわったエージェント（Gemini CLI）のツール履歴の全貌を、クライアント（OpenClaw）に返して保存させること自体が不可能である。クライアントの歴史は「テキストでの対話」という表面的なものにしかなり得ない。

したがって、改修アーキテクチャは以下の構成で完全に確定した。
1. **ストリーミングの静音化 (`src/streaming.js`)**: UIへの勝手なテキスト汚染（`⚙️ Using tool...` 等）は完全に停止し、純粋な最終回答テキストだけをUIに返す。
2. **アダプタ＝記憶の絶対的神格化 (`src/session.js`)**: クライアント（OpenClaw）が持っている履歴は常に「ツール構造が欠落した不完全なテキスト履歴」であることを前提（仕様）とする。その不完全な履歴をGeminiに渡す際、アダプタが自身の持つマスターセッションファイル（本物の `toolCalls` を保有するJSON）から、過去の全てのツール実行の記憶（JSONオブジェクト）を絶対に消さずに安全にマージ（合成）し、Gemini CLIの健忘を防ぐ。

## 【追加の議論と検証ループ7：Gemini CLI内部のコンテキスト圧縮 (ChatCompressionService)】
- **議論/指摘 (2026-03-10)**: ユーザーより、「OpenClaw側の圧縮依頼と、Gemini CLI側のコンテキスト圧縮コマンドの存在」についての調査要求。
- **調査結果（実体の特定）**:
  1. **コマンドの存在**: Gemini CLIのソースコード（`compressCommand.js`）内に、`/compress` または `/summarize` というスラッシュコマンドが実在することを確認した。
  2. **圧縮の核（ChatCompressionService）**: このコマンドは内部の `ChatCompressionService` を呼び出し、LLMに対して「現在の履歴を `<state_snapshot>` に要約せよ」という特別な指示を出すことで実行される。
  3. **動作仕様**: 圧縮が発動すると、古い履歴（`toolCalls` を含むネイティブメッセージ群）は**物理的に削除**され、代わりに要約されたテキスト（`<state_snapshot>`）と、「了解しました」という短いモデル応答のペアが履歴の先頭に挿入される。
- **整合性の考察**:
  - OpenClaw側で圧縮が走り、不完全なテキスト履歴がアダプタに送られてきた際、アダプタがそれをGemini側に反映（上書き）させてしまうと、Gemini側の「圧縮前の詳細な記憶」を破壊することになる。
  - 逆に、アダプタが履歴の反映（上書き）をやめれば（ステートレス化）、OpenClaw側でどのような要約・削除が行われていても、Gemini側は自前の完璧な（あるいは自前で圧縮済みの）履歴を `--resume` でロードし続けるため、整合性の崩壊は起きない。

**【結論の補強】**
Gemini CLI自体が「自律的に記憶を整理・要約する能力（`<state_snapshot>`）」を内包している以上、外部（OpenClaw/アダプタ）がその記憶を操作しようとすることは、常に情報の欠落やハルシネーションのリスクを伴う。したがって、アダプタが履歴の責任を手放し、リレーに徹する「ステートレス・リレー（SSoT）戦略」は、コンテキスト圧縮の観点からも最も安全かつ合理的な設計であるといえる。

## 【追加の議論と検証ループ8：圧縮（要約）プロンプトのインターセプト】
- **議論/指摘 (2026-03-10)**: ユーザーより、「OpenClaw側がLLMに投げる『圧縮用プロンプト（要約依頼）』をアダプタが検知し、Gemini CLIを動かさずに『Gemini側で自動で行われます』と即答（インターセプト）すれば、履歴の破壊を完全に防げるのではないか」という極めて実戦的な解決策の提案。
- **考察と整合性**:
  1. **二重要約の弊害**: OpenClawがテキストベースで要約を行うと、Gemini CLIが保持している「詳細なツール実行履歴」や「`<state_snapshot>` による高品質な要約」が、情報の薄い「テキストの塊」で上書きされてしまう。
  2. **インターセプトの利点**: アダプタが「要約リクエスト」と思われる特定のJSON構造やプロンプトを検知して遮断することで、OpenClaw側には「要約に成功した」と思わせつつ、Gemini側のマスター履歴（SSoT）は一切汚染させずに守り抜くことができる。
- **最終解決パラダイムの確定**:
  - **履歴の上書きを廃止 (SSoTの確立)**
  - **UI装飾テキストの排除 (ストリーミングの静音化)**
  - **圧縮依頼のインターセプト (外部干渉の遮断)**

**【結論の総括】**
Gemini CLIの多段自律エージェントという性質を維持するためには、外部クライアントによる履歴の「整理」操作を無効化し、Gemini CLIの内部ロジックに100%委ねるのが正解である。アダプタは「リレー」と「ガード（干渉のインターセプト）」に徹することで、システムの堅牢性を最大化する。
