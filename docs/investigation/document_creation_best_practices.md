# 調査報告書: ソフトウェアドキュメンテーションのベストプラクティスと自動化の要件

**調査方針（Strict-Investigator Rules Applicable）**
本レポートは、AIエージェントによるドキュメント自動生成スキルを作成するための事前調査として、客観的な事実（Webや主要IT企業のガイドライン）のみを収集・抽出したものである。ここに推測や実装の提案は含めない。

## 1. 業界標準のドキュメント構成フレームワーク: Diátaxis (ディアタクシス)
現在、Canonical (Ubuntu) をはじめとする多くの技術ドキュメントで採用されている4象限の事実上の標準フレームワーク。情報を「ユーザーの目的」に応じて完全に分離することが求められる。

### Diátaxis の4区分
1. **Tutorials (学習指向)**
   *   **目的**: 初学者が「手を動かして何かを達成し、学ぶ」ためのガイド。
   *   **ルール**: 概念の説明は最小限に留め、必ず成功する具体的な手順のみを記載する。
2. **How-To Guides (問題解決・タスク指向)**
   *   **目的**: ユーザーが特定のゴール（「〇〇をしたい」）を達成するためのステップバイステップの手順。
   *   **ルール**: 前提知識をある程度仮定し、チュートリアルよりも広範だが、理論には踏み込まない。（例: Runbook, 障害対応）
3. **Reference (情報指向)**
   *   **目的**: システムの正確な仕様や構造の提供。
   *   **ルール**: 辞書のように網羅的で正確であること。コード、API、スキーマ、アーキテクチャと1対1で対応する。（例: API仕様書、アーキテクチャ図）
4. **Explanation (理解指向)**
   *   **目的**: 対象をより深く理解するための背景知識の提供。
   *   **ルール**: 手順は含めず、「なぜそのように作られたか」「歴史的な経緯は何か」を論じる。（例: ADR, 調査レポート、Devlog）

## 2. ライティングとフォーマットのルール (Google Developer Documentation Style Guide 基準)
Googleが公開しているデベロッパー向けドキュメントのスタイルガイドから、AIによる生成時に強制すべき事実基準を抽出。

*   **トーンと語り口**: 会話的（conversational）で友好的であること。不必要に難解な言葉（jargon）や無駄な装飾（frivolousな表現）を避ける。
*   **文法と視点**: 二人称（"you"）を用い、能動態（active voice）で記述する。
*   **条件節の配置**: 操作指示を行う際は、条件を先出しする。（例: ✕ "Click X if you want to Y." → 〇 "If you want to Y, click X."）
*   **フォーマットの一貫性**: 
    *   タイトルや見出しはセンテンスケース（先頭のみ大文字）を使用する。
    *   順序のある手順は「番号付きリスト」、順序を問わない列挙は「箇条書き（バレット）リスト」を厳格に使い分ける。
    *   コード内の要素は等幅フォント（バッククォート）、UI要素は太字で強調する。
*   **Minimum Viable Documentation**: 必要最小限で、常に新鮮（fresh）で正確な情報を維持する。古びた情報は「無い方がマシ」とされる。

## 3. AI特有のコンテキスト要件 (MCP / LLM 向け仕様)
AIエージェント自身がコードベースを維持・理解するためのメタ的なドキュメント要件に関する事実。

*   **機械可読性の重視**: LLMがパースしやすいよう、Markdown (MDXや標準Markdown) 形式が好まれる。
*   **LLM向けのコンテキスト注入 (SSoT)**: 人間向けではない「AIへの絶対的な指示」や「過去の泥臭いハックの理由」を集約した単一の真実源 (Single Source of Truth) ファイルを用意し、エージェント実行時のシステムプロンプトや初期コンテキストに強制的にロードさせる運用がベストプラクティスとなっている。
*   **メタデータの分離**: スキルやツールの仕様は `SKILL.md` のような標準化されたフロントマター（YAML）＋Markdownで記述され、MCPサーバーから動的に提供される。

## 4. 自動化スキル構築のための要件定義（事実からの導出）
以上の事実から、自律型ドキュメント生成スキルを構築する上で満たすべきステップは以下のようになる。

1.  **入力取得**: 対象コードベースの特徴、過去のチャットログ（Devlog相当）、既存のメモ等をすべて走査・パースする。
2.  **Diátaxis分類**: 取得したコンテキストを「Tutorial」「How-To」「Reference」「Explanation」の4象限に厳密に振り分ける。
3.  **スタイル適用（Google基準）**: 能動態、二人称、UIの太字化、コード要素のバッククォート囲み、条件の先出しをプロンプトで強制する。
4.  **AI向けSSoTの分離確認**: 「AI_CRITICAL_CONTEXT.md」のような特殊なファイルへの干渉を避け、人間向けの階層とAI向けの階層を分けて保存・出力する。
