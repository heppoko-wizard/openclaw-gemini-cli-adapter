# gogcli 認証メカニズムと最小権限の技術調査レポート

## 1. 調査目的
`gogcli` において、`--services` を省略した際になぜ意図しない大量の権限が要求されるのか、および「ダミーサービスを指定する」以外のスマートな回避策（公式なベストプラクティス）が存在するかを、ソースコードレベルで特定する。

## 2. 調査結果 (事実)

### A. デフォルト動作の強制 (ソースコード L31, L106@auth_add.go)
`gog auth add` コマンドにおいて、`--services` 引数が未指定（空）の場合、gogcli は内部で以下の分岐を通ります。
- `ServicesCSV` の初期値は `"user"` である。
- `internal/cmd/auth.go` の `parseAuthServices` 関数内にて、`""`, `"user"`, `"all"` のいずれかが検知されると、強制的に `googleauth.UserServices()` が呼び出される。
- `UserServices()` は、Gmail, Drive, Chat, Classroom を含む **全13種類のサービス名** を返却する。

この挙動を無効化する隠しフラグや設定オプションは**存在しません。**

### B. 権限の混合要求ができない仕様
gogcli は「サービス単位」で権限をパッケージ化してリクエストする設計になっています。
- `--readonly` フラグは **グローバル（全サービス共通）** に作用する。
- 例：カレンダーはフル権限、スプレッドシートは参照専用にしたい場合、gogcli の標準フラグ（`--services calendar,sheets --readonly`）では両方とも参照専用にされてしまい、片方だけフル権限にすることは不可能。

### C. 公式ドキュメントの記載 (`docs/spec.md` L122)
公式スペックには以下のように明記されています。
> "To request fewer scopes, choose fewer services via `gog auth add --services ...`"
（より少ないスコープを要求するには、`--services` でより少ないサービスを選択せよ）

しかし、前述の通り「0個」の選択は「全選択」に化けるため、技術的に「最低1つ」のサービス選択が必須となっています。

## 3. なぜ「ダミーサービス」が必要なのか
`gogcli` のパーサーをエラー（`unknown service`）にせず、かつ内部の「全13サービス自動追加」ロジックを回避するための唯一の道は、**「実害がない（または既に要求に含まれている）有効なサービス名を1つだけ明示的に渡す」** ことです。

- **people** サービスは、標準的な `profile` スコープ（ユーザー名、写真等）のみを要求します。これは GCP プロジェクトで定義済みの最小限の非機密情報であり、他のサービス（Gmail等）のような破壊的権限を持ちません。

## 4. 結論
gogcli の現在の実装において、GCPプロジェクトの定義と 100% 一致する最小権限を実現するための最も「正攻法」かつ「技術的に唯一」のパラメータ構成は以下の通りとなります。

```bash
gog auth add <email> \
  --services people \
  --extra-scopes <GCPで定義した正確なスコープリスト>
```

事実として、この「めんどくさい」パラメータ設計以外に、gogcli のデフォルト挙動を抑制してカスタムスコープのみを通す公式な手段は存在しないことが確認されました。
