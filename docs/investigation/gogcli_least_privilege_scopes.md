# gogcli Least Privilege Scopes 調査レポート

## 調査目的
`gogcli` (steipete/gogcli) において、不要なサービス（Gmailなど）の強大なスコープを含めずに、GCPプロジェクトで定義された最小限の非機密スコープのみを要求するための「公式な仕様・ベストプラクティス」をソースコードおよび公式ドキュメントから事実確認する。ユーザーの「ダミーサービスを指定するようなハック（めんどくさいこと）が本当に最適解なのか」という疑問に答える。

## WEBおよび公式ドキュメントから得られた事実
1. **gogcli の設計思想**: `gogcli` は特定のGoogle Workspace API（Gmail, Drive等）を操作するためのCLIツールであり、何らかの「サービス」を操作することを前提に作られている。
2. **公式な最小権限アプローチ**: 公式のベストプラクティスとして案内されているアクセス制限（Least Privilege）の方法は、「対象のサービスを指定した上で、`--readonly` フラグや `--gmail-scope readonly` / `--drive-scope file` を併用する」というアプローチである。
3. **無（ゼロ）サービスのサポート**: ドキュメントには「サービスを何も指定せずに、拡張スコープ(`--extra-scopes`)のみで認証を通す」というユースケースはベストプラクティスとして記載されていない。

## ソースコードから得られた事実
前回の調査で確認した通り、`internal/cmd/auth.go` の `parseAuthServices` 関数には以下の処理がハードコードされている。

```go
func parseAuthServices(servicesCSV string) ([]googleauth.Service, error) {
	trimmed := strings.ToLower(strings.TrimSpace(servicesCSV))
	// 引数がない ("")、または "user", "all" の場合は、全ユーザーサービスを返す
	if trimmed == "" || trimmed == "user" || trimmed == literalAll {
		return googleauth.UserServices(), nil
	}
// ...(後略、カンマ区切りでパースし、未知の文字列があればエラーを返す)
```

1. **空指定（引数なし）の挙動**: `--services=""` と指定したり、引数自体を省略すると、コード上強制的に `UserServices()`（全13サービスの詰め合わせ）が返却される。これを防ぐ設定ファイルや環境変数は存在しない。
2. **無効な文字列の挙動**: `"none"`, `"dummy"`, `"empty"` などの存在しないサービス名を与えると、`errUnknownService` 例外がスローされ、コマンド自体が失敗する。
3. **回避策の必要性**: したがって、全サービスのデフォルトスコープ展開を回避し、かつコマンドを成功させるには**「`gogcli` が知っている（エラーにならない）有効なサービス名の中で、最も特権を持たない安全なサービス」**を最低1つ指定してパーサーを通過させるしかない。

## 結論
事実として、`gogcli` のソースコードレベルで「ダミーサービス（例：`people` - スコープは `profile` のみ）を指定する」というアプローチが、**全スコープの強制追加を回避するための唯一の技術的解決策**であることが裏付けられた。

公式の機能である `--extra-scopes` と、最も影響の少ない既存のサービス `people` の組み合わせは、「ハック」のように見えて、現在の `gogcli` のパーサー仕様を正攻法で満たすための最短ルートとなる。
