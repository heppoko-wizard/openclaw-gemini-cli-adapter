# 調査レポート: ハイブリッド履歴同期基盤 (SSoT 3.0) のアーキテクチャ設計と検証

## 1. 調査の背景と目的
OpenClaw (フロントエンド) と Gemini CLI (バックエンド) をつなぐ Adapter において、UI 用に進捗を表示するプレースホルダーテキスト（例: `⚙️ Using tool [cron] ...`）が履歴汚染を引き起こし、AI が幻覚（ハルシネーション）を起こす問題が再発していた。
この汚染問題を根本的に解決し、かつ Gemini が要求する厳密な構造化データ（`toolCalls`, `functionResponse`）を破綻させずに同期するアーキテクチャの設計を目的とする。

## 2. 仮説・調査・検証のプロセス（議論の変遷）

### フェーズ 1: ブラックリスト（フィルター）による除外案
- **初期仮説**: Adapter が履歴を Gemini に送る直前で、単に `⚙️ Using tool [cron]` 等の UI 用テキストを `filter` 等で除去すれば良いのではないか。
- **ユーザーからの指摘**:
  - 「それでは、ツールの実行履歴そのものが削除されることになり、Gemini は『ツールを使わずに無言で結果を得た』と学習（幻覚化）してしまうのではないか」という極めて重要な指摘を受けた。
- **検証結果（事実の確認）**:
  - 指摘通り、Gemini API は厳密な関数呼び出しの実績（`functionCall` と `functionResponse`）がコンテキストに残っていないと、過去の自身の振る舞いを正しく解釈できない仕様であることが確認された。
  - したがって、単純なテキストの除去（フィルタリング）は解決策として**誤り**であると判断された。

### フェーズ 2: コンバーター（1:1翻訳）チェーンの適用案
- **新たな仮説（ユーザー提案）**:
  - 「OpenClaw（OpenAI API形式）の履歴をGemini形式にマッピングする `src/converter.js` を `src/streaming.js` に適用すれば、ツールコールのメタデータが正しく1:1で翻訳され、問題が解決するのではないか」。
- **調査と事実の確認**:
  - `src/converter.js` 内に `convertToGeminiMessages` という完全な1:1翻訳関数が存在する事実を確認した。
  - しかし、**「ツールの実行主体が（OpenClawではなく）Gemini CLI側にある」** ため、Adapter は OpenClaw フロントに対して構造化データではなく、（UI表示用の）「ただのテキスト」しか流し込んでいなかった。
  - そのため、OpenClaw側の履歴上ではツール実行が「純粋なテキストの会話（AIのセリフ）」として記録されており、いくら完璧な1:1翻訳を適用しても、コンバーターはそれを「アシスタントの発言テキスト」として忠実に翻訳してしまう事実が判明した。

### フェーズ 3: 外部状態保持（DB化）案とその限界
- **代替案**:
  - Adapter 側で独自に `adapter-history/<sessionKey>.json` のような状態（State）を持ち、Gemini に流し込む直前で UI のプレースホルダーテキストをメタデータ（構造体）に「代入・復元」するアプローチ（SSoT 2.0）。
- **ユーザーからの更なる指摘**:
  - 「（Adapter に状態を持たせるのではなく）そもそも `tool_use` や `tool_result` をストリーミングしてきた構造そのものを、会話履歴として合成（UI に埋め込み）し、パースしやすいように加工すればよいのではないか」。

### フェーズ 4: 「インライン・メタデータ展開方式（SSoT 3.0）」の検証（最終解結）
- **仮説（インライン方式）**:
  - Adapter 側で記憶を持たず（ステートレス）、GeminiCLI からストリーミングされてくる JSON 구조体を特定のマーカー（または HTML のコメント `<!-- -->`）で囲んで OpenClaw にテキストとして送る。
  - 次のターンで OpenClaw から送られてくる生テキストからその JSON を抽出し、復号化（代入）して Gemini へ渡す。
- **フィージビリティ（実現可能性）の技術調査**:
  1. **復号化に必要な情報が完全に揃っているか？**
     - `@google/gemini-cli-core/dist/src/output/stream-json-formatter.test.js` などを調査した結果、ストリーミングされる JSON（`tool_use` や `tool_result`）には `tool_id`, `tool_name`, `parameters`, `status` などの完全なメタデータが含まれていることが確認された。
  2. **`tool_id` の一意性と紐付けの信頼性**
     - ソースコードおよび Gemini API ドキュメントの調査から、`tool_id`（`functionCall.id`）は「ツール名固定のID」ではなく、「実行ごとの完全なランダム文字列（24文字程度のユニークID）」であることが判明した。これにより、順序や複数回呼び出しを問わず、100% 正確に履歴の紐付けと再構築が可能と判断された。
  3. **UI 汚染問題への対処**
     - JSON メタデータをそのまま送ると UI が文字列で埋め尽くされる問題に対して、OpenClaw（Markdown レンダラー）の特性を利用し、`<!-- <tool_metadata>{...}</tool_metadata> -->` と HTML コメント化して隠蔽（不可視化）する設計を立案した。

## 3. 結論と最終アーキテクチャ (SSoT 3.0)
一連の議論と調査を通じ、**「 Adapter 自身は状態を持たず、UI 送信時のテキストの背後（HTMLコメント内）に JSON 構造を潜ませ、次ターン送信直前でそれを正規表現で抽出し、Gemini の構造化データとして復元・アタッチする（インライン展開方式）」** が、最も堅牢でバグを生まない最終的なアーキテクチャ（真の Single Source of Truth）であると結論づけられた。

## 4. 実行タスク
本調査結果に基づき、以下のファイルに対する改修計画（Implementation Plan）を確定させ、実装フェーズへ移行する。
- `src/streaming.js`
  - メッセージ抽出・再構築ロジックの刷新（`<tool_metadata>` マーカーのパースと `toolCalls` ヘのマッピング）。
  - ストリーミング結果（`tool_use`, `tool_result`）への HTML コメント化された JSON 埋め込み処理の追加。
