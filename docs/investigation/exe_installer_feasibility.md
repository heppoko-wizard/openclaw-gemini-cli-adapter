# 調査レポート：Node.js スクリプトの .exe 化によるインストーラー作成

**調査日時**: 2026-03-10
**調査対象**: `interactive-setup.js` および `openclaw-gemini-cli-adapter` 全体のバイナリ化

---

## 【調査の道筋（仮説と検証のトレイル）】

### ステップ1：Node.js 未インストール環境での実行可能性
- **仮説**: Node.js ランタイムがない Windows 環境でも、`.exe` ファイル単体でセットアップを完結させたい。
- **検証**: Node.js 標準の **SEAs (Single Executable Applications)** およびサードパーティツールの **pkg**, **nexe** の仕様を確認。
- **事実の発見**: 
  - SEAs (Node.js 19.7.0以降): Node.js 本体のバイナリに JS コードと設定を注入して単一の `.exe` を作成可能。
  - pkg: 以前から広く使われているツールで、Node.js ランタイムごとバンドルして `.exe` を出力可能。
  - これらにより、**ユーザー環境に Node.js がなくても JS スクリプトを起動すること自体は可能**。

### ステップ2：依存パッケージ（node_modules）の取り扱い
- **仮説**: `@google/gemini-cli` など、複雑な依存関係を持つパッケージも `.exe` に含めることができるか。
- **検証**: 各バイナリ化ツールの資産取り込みロジックを確認。
- **事実の発見**: 
  - `pkg` 等のツールは、静的解析によって `require()` されているファイルを抽出し、仮想ファイルシステム（snapshot）内に同梱する。
  - ただし、C++ アドオンなどネイティブモジュールが含まれる場合、`.exe` の外に個別のバイナリを置くか、実行時に一時ディレクトリに展開する必要がある。

### ステップ3：外部コマンド呼び出し（spawn）の制限
- **仮説**: `.exe` 化したスクリプトから `spawn('npm', ...)` や `spawn('tailscale', ...)` を呼ぶ際に問題はないか。
- **検証**: 既存の `interactive-setup.js` 内のパス解決ロジックを再確認。
- **事実の発見**: 
  - `.exe` 自体は起動するが、スクリプト内で `spawn` している外部コマンド（`npm`, `git`, `winget`, `tailscale`）は **システムにインストールされている必要がある**。
  - 「`.exe` ひとつですべてを完結」させるには、これら外部ツールのポータブル版を同梱するか、スクリプト内でそれらを自動ダウンロード・パス設定するロジックが必要。

## 【コードベース・仕様から確認された最終的な事実】

| 項目 | 実現可能性 | 備考 |
|---|---|---|
| **Node.js ランタイムのバンドル** | ◯ 可能 | SEAs または `pkg` を使用。 |
| **JS ソースコードの隠蔽** | ◯ 可能 | V8 スナップショットやバイトコード化により、難読化・保護される。 |
| **1つの .exe によるセットアップ** | △ 条件付き | Node.js は不要になるが、`npm` 経由のインストールを行う場合、`npm` 自体（もしくは Node.js SDK 全体）をセットアップスクラプトが自前で展開する必要がある。 |
| **既存コードの修正** | ◯ 必要 | `__dirname` などのパス参照を、バイナリ実行時（仮想ファイルシステム）でも正しく動くように調整（`process.cwd()` や `path.dirname(process.execPath)` 等）する必要がある。 |

## 【追加調査：実行形式ごとの環境制限の比較】

ユーザーの懸念に基づき、形式ごとの「実行の通りやすさ（不確実性）」を整理しました。

| 形式 | 主な実行制限 / ブロッカー | 特徴と事実 |
|---|---|---|
| **.bat** | Group Policy, SmartScreen | ほぼ全ての Windows で動作するが、管理者によって CMD 自体が禁止されている場合や、セキュリティソフトが不審な挙動として検知する場合がある。 |
| **.ps1** | **ExecutionPolicy** | デフォルトでは `Restricted`（実行禁止）に設定されていることが多く、スクリプト実行前にポリシーの変更が必要。企業環境では GPO で固定され変更不可の場合がある。 |
| **.exe** | **SmartScreen**, Antivirus | ポリシー（ExecutionPolicy）の影響は受けないが、**「電子署名（Developer Certificate）」**がない場合、SmartScreen が「Windows によって PC が保護されました」という警告を出し、実行がブロック（または「詳細」をクリックして実行）される可能性が高い。 |

### 事実のまとめ
- **.ps1** は最も制限に当たりやすく、実行のためにユーザーによるポリシー変更やフラグ（`-ExecutionPolicy Bypass`）が必要になる可能性が恒常的にあります。
- **.exe** はポリシーによる制限を回避できますが、署名のない独自ビルドの `.exe` の場合、セキュリティソフトや SmartScreen による「信頼性の欠如」を理由としたブロックが発生しやすいという不確実性があります。
- いずれの形式も「100% どこでも確実に動く」わけではなく、それぞれの仕組みに応じた「信頼性（署名）」や「ポリシーのバイパス」の考慮が必要です。

## 【最終比較：開発コスト vs 実行安定性】

どの形式が「最も簡単に、エラーなく」運用できるかの事実比較です。

| 形式 | 開発の容易さ (作成コスト) | 実行の安定性 (環境トラブル) | 総合的な事実 |
|---|---|---|---|
| **.bat** | **◎ 最も容易**。メモ帳でも書け、特別なビルドツールは不要。 | **△ 低い**。変数の展開失敗や文字化けなど、Windows のレガシーな仕様に起入するエラーが起きやすい。 | 作成は早いが、環境ごとの微調整に最も時間がかかる。 |
| **.ps1** | **◯ 容易**。Node.js に近い柔軟な記述ができる。 | **× 要注意**。OS のデフォルト設定でブロックされるため、ユーザーが解除操作を行わない限り「動かない」というエラーになる。 | コードは綺麗に書けるが、実行までのハードルが最も高い。 |
| **.exe** | **△ やや困難**。`pkg` や `nexe` 等のツール導入とビルド工程が必要。 | **◯ 高い（条件付）**。Node.js の有無を問わない。ただし署名がないと SmartScreen の警告（エラーに見える表示）が出る。 | ビルド環境の構築に手間はかかるが、実行時のランタイム依存エラーは最小限に抑えられる。 |

## 【追加調査：事前に想定可能な Windows 固有エラーと対策】

「変なエラー」の多くは、Windows の古い仕様やセキュリティ仕様に起因しており、以下の手法で事前にかなりの部分を防御可能です。

### 1. バッチファイルの「変数展開」エラー
- **事実**: `if` や `for` 内で変数を書き換えると、通常は「実行前」の値で固定される。
- **対策**: `SETLOCAL ENABLEDELAYEDEXPANSION` を宣言し、`%var%` ではなく `!var!` を使うことで、実行時の最新値を参照できる。

### 2. 文字化け（Mojibake）エラー
- **事実**: 日本語 Windows は標準で Shift-JIS (CP932) を使うが、最近のツールは UTF-8 (CP65001) を前提とする。
- **対策**: 
  - バッチファイルの冒頭に `chcp 65001 >nul` を入れ、コンソールのエンコーディングを UTF-8 に強制する。
  - ファイル自身を「BOMなし UTF-8」で保存する（BOMがあると cmd.exe が誤動作する事実がある）。

### 3. パスの「空白」エラー
- **事実**: `C:\Program Files\...` のように空白があると、引数が分割されてエラーになる。
- **対策**: 
  - パスを扱う変数は常に `"%VAR%"` のように二重引用符で囲む。
  - `start` コマンドを使う際は、最初の引数に空のタイトル `start "" "C:\Path..."` を入れる（最初の引用符をタイトルと見なす仕様への対策）。

### 4. PowerShell の「実行ポリシー」エラー
- **事実**: `.ps1` はデフォルトで実行が禁止されている。
- **対策**: バッチファイルから呼ぶ際に `powershell -ExecutionPolicy Bypass -File ...` のように明示的にポリシーを回避するフラグを渡す。

### 5. Node.js `spawn` 時の `ENOENT` エラー
- **事実**: Windows では `.bat` ファイルは `shell: true` オプションを付けないと「ファイルが見つからない」エラーになる。
- **対策**: `child_process.spawn(..., { shell: true })` を徹底する。

### 結論としての事実
1. **開発が最も簡単なのは `.bat`** ですが、実行時の「予期せぬ挙動（エラー）」は最も多くなります。
2. **実行時の環境起因エラーを最小化できるのは `.exe`** ですが、開発（ビルド）の手間と SmartScreen 対策という別の課題が生じます。
3. **`.ps1`** は記述は容易かつ強力ですが、標準のセキュリティポリシーが最大の壁となります。

## 【追加調査：バッチファイル（.bat）による Node.js / Bun の自動インストール】

「.sh でできていることが .bat でできるか」という問いに対する事実は以下の通りです。

| 対象 | 実現可能性 | 実行方法（事実） |
|---|---|---|
| **Node.js** | **◯ 可能** | `winget install OpenJS.NodeJS.LTS` をバッチから実行可能。最近の Windows 10/11 には標準搭載されている。 |
| **Bun** | **◯ 可能** | バッチから `powershell` を呼び出し、`irm bun.sh/install.ps1 | iex` を実行することでインストール可能。 |

### .sh との違い
- **.sh**: `curl ... | bash` のように、「ワンライナー」でパイプして即座に実行する文化が強い。
- **.bat**: 自前で全ロジックを持つのは困難だが、`winget` や `powershell` という「外部の強力なツール」を呼び出すコマンドハブとして機能する。

### 実現上の制約
- **反映のタイミング**: `winget` 等でインストールした直後は、同じパス（環境変数）が反映されないため、「一度ターミナルを閉じさせる」か、スクリプト内で `refreshenv` に相当するパス再読み込みロジックを組む必要がある（.sh でも同様の制約がある場合が多い）。
- **winget の有無**: 非常に古い Windows 10 や Windows Server 環境では `winget` がないため、その場合は `curl` 等で直接バイナリを落とすロジックが必要になる。

---

## 【結論】
**「.exe ファイルとしてすべてのインストールを完了すること」は技術的に可能です。**

ただし、現在のアダプタが `npm install` や `winget` に依存しているため、完全に「何も入っていないクリーンな Windows」で実行する場合、セットアップ `.exe` が以下の役割を担う必要があります：
1. 自分の中に Node.js/npm バイナリを（データとして）持ち、一時的に展開して実行する。
2. もしくは、`winget` 等を利用してシステム側に必要なものを揃えたあと、自身の中身を展開する。
