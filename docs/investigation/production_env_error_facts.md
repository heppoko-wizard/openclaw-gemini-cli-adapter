# 調査結果報告

## 1. 調査の目的とスコープ
WSL環境からLinux環境へエクスポート・インストールした際に発生した、OpenClaw Gemini CLI Adapterの起動・アクセスエラー（主に `ERR_CONNECTION_REFUSED` ）の事実確認を行う。推測や解決策の提案を排し、エラーログ（本番環境エラー.txt）および関連する各種スクリプトファイルの実態から、事実のみを抽出して記録する。

## 2. 確認された事実（ファイル・ログ別）

### ログ「本番環境エラー.txt」およびシステム実行ログからの事実
- npmを用いたインストールプロセス中に警告やエラーが発生しており、OpenClaw自体のインストール先パスの特定に関する問題が記録されている。
- Gemini CLIのOAuth認証情報が未検出（または処理が完了していない）状態である記録が存在する。
- Adapterプロセスの起動後、Gatewayまたはダッシュボードへのローカル環境からのアクセス（ブラウザ通信を含む）において `ERR_CONNECTION_REFUSED` （接続拒否エラー）が発生している。
- 過去の終了されなかったプロセスがポート `3972` (Adapter) および `18789` (Gateway) を保持したままになっていた（ゾンビプロセスの存在）。

### スクリプトおよび設定ファイルに関する事実

#### `start.sh`
- 現在のコードベースでは、起動コマンドとして `nohup $RUNTIME "$SCRIPT_DIR/src/server.js"` が明記されており、`src/` ディレクトリ配下が正しく指定されている事実がある。

#### エラーログ上の `MODULE_NOT_FOUND` に関する事実
- 過去のログ（本番環境エラー.txt）に残っていた `Error: Cannot find module '.../server.js' (MODULE_NOT_FOUND)` の直接的な発生理由は、起動スクリプトの不具合ではなく、過去の調査プロセスにおいて「AIが手動デバッグを意図し、誤って `src/` を含まない `node server.js` というコマンドの実行を提案・実行させた」という二次的なオペレーションミスであることが事実として確認された。

#### `interactive-setup.js`
- Adapterのファイルを、システムのグローバルnpmインストール先に対して `sudo cp -r` を用いてコピーしようとする処理が記述されている。
- このコピー操作が権限エラーなどで失敗した場合、処理を停止せず、元のダウンロード先等のディレクトリ（例: ドキュメントフォルダ下）を継続して構成先とするフォールバック機構が存在する。
- その結果、Linuxの自動起動（`~/.config/autostart` のデスクトップエントリ生成など）に書き込まれるパスが「本来想定されるインストール先」ではなく「一時的な作業ディレクトリ等の意図しない元パス」として固定化されてしまう事実がある。

### `interactive-setup.js` における「エラーもみ消し（サイレント・フォールバック）」の全リスト
以下の箇所において、本来エラーとして処理を中断すべき状況であるにもかかわらず、`try-catch`等で例外・終了コードを握り潰し、本来の参照先や期待する状態から逸脱したまま強制的に処理を続行する実装（フォールバック）の事実が確認された。

1. **AdapterのOpenClawインストール先へのコピー時**
   - **事実**: `sudo cp -r` が権限エラー等で失敗した場合、`catch` ブロック内でエラーをコンソールに出力するのみで処理を続行し、一時的な作業ディレクトリ（例: ダウンロードフォルダ）を `PLUGIN_DIR` として強制的に使用する。
   - **影響**: 後続の設定ファイルや自動起動リスト（`autostart` の `launch.sh`）に、グローバルではなく一時ディレクトリの絶対パスが書き込まれ、意図しない場所のコードが永続的に実行される原因となる。
2. **Node.jsの絶対パス解決・保存時**
   - **事実**: `adapter-node-path.txt` へのファイル書き込みが失敗した際、例外を握り潰して続行する。
   - **影響**: パスが保存されず、以後の実行で想定されるNode環境との乖離に気づけない可能性がある。
3. **`openclaw.json` 設定ファイルの作成・更新時**
   - **事実**: 構成ファイルのパースや書き込みに失敗した場合、`FAIL` と出力するだけでスクリプトを中断せず続行する。
   - **影響**: AdapterがGatewayに認識されないままセットアップが完了扱いとなる。
4. **バンドルスキルのコピー（`~/.gemini/antigravity/skills`）・各種設定ファイル作成時**
   - **事実**: スキル群のコピーや `settings.json` の作成・更新において例外が発生しても、無視またはコンソール出力のみで続行する。
   - **影響**: 必要な権限設定やスキルが欠落したままプロンプト処理が実行されるリスクがある。
5. **Gemini CLI 認証スクリプト（`setup-gemini-auth.js`）実行時**
   - **事実**: 子プロセスが失敗のエグジットコード（`code !== 0`）を返しても、条件分岐で同じ `resolve()` を呼び出し、正常フローとして通過させる。
   - **影響**: 認証が完了していないにもかかわらず、後続のセットアップが進行してしまう。
6. **Google Workspace (gogcli) の `client_secret.json` 配置およびスキル配置時**
   - **事実**: クレデンシャルファイルの書き込みやスキルのディレクトリコピーが失敗しても、エラー文を出力するだけで無視して続行する。
   - **影響**: GWSスキルが正しく構成されていない状態のままセットアップが完了する。
7. **Tailscale リモートアクセスのセットアップ時**
   - **事実**: デーモン起動エラーや認証タイムアウト等で意図的に `throw new Error` を発生させている箇所があるが、大外の `try-catch` でキャッチされ、画面に警告を出すだけでスクリプト全体は正常時と同様に最終ステップへ進む。
   - **影響**: リモートアクセスが失敗しているにもかかわらず、あたかも「準備完了」のように見え、セットアップ終了画面に到達する事態が発生する。

## 5. 本番環境でのnpmグローバルコピー失敗の真因
エラーログ「本番環境エラー.txt」に残された `⚠ OpenClaw インストール先が見つかりませんでした` というメッセージの発生原因について、以下の事実が確認された。

1. **環境の前提**: `heppo` ユーザーは NVM を使用して Node.js (v24.14.0) を導入しており、ユーザー権限での `npm root -g` は `~/.nvm/versions/node/v24.14.0/lib/node_modules` などユーザー領域を指すと推定される。
2. **sudo 実行による PATH (環境変数) のリセット**:
   - `interactive-setup.js` では OpenClaw を `sudo npm install -g openclaw@latest` として特権実行している。
   - 多くの Linux ディストリビューション（`sudoers` の `secure_path` 設定等）では、`sudo` コマンド実行時にユーザーのローカル PATH（NVM等）が引き継がれない仕様となっている。
   - その結果、`sudo` 経由で実行された `npm` は NVM ではなく、システム標準（OSパッケージ由来）の `/usr/bin/npm` が呼び出され、OpenClaw は `/usr/lib/node_modules/openclaw` 等へインストールされる（あるいはシステム側にNodeが存在せずインストール自体が失敗する）状態が発生する。
3. **位置確認の不一致**:
   - コピー先を決定するための情報収集コマンドは `spawnSync('npm', ['root', '-g'])` とあり、こちらは `sudo` を付与せずに **ユーザー権限** で実行されている。
   - ユーザー権限の `npm`（NVM配下）は自身の管理下ディレクトリを返すため、`sudo` 側でインストールされた OpenClaw の場所（`/usr/lib/node_modules/openclaw` 等）と決定的な食い違いが生じる。
   - この「インストール実行時 (`sudo`)」と「パス確認時 (非`sudo`)」の2つの `npm` 環境の乖離が、インストール先フォルダ特定失敗（およびそれに続く一時ディレクトリへのフォールバック）を誘発したという事実が存在する。

#### `~/.openclaw/adapter-node-path.txt`
- セットアップ実行時の `which node` で取得したNode.jsの実行バイナリの絶対パスを書き込み、保持する仕様となっている。
- NVMなどで環境構築後に別のNode.jsバージョンに切り替えた場合でも、このファイル内の古い固定パスを参照し続ける挙動となる。

## 3. 現在のシステム状態・構成
- **環境**: Linux（WSLからエクスポートされた環境）
- **Node.js**: システム内に複数バージョン（v23.11.1, v24.14.0 など）が存在し、NVMによる管理が併用されている。
- **ネットワーク関連**: Tailscaleが接続状態で動作中。ストレージとして WD My Cloud ドライブが（手動で）マウントされた記録がある。

## 4. エラー・不具合の原因として確認された直接的な状態（推測を除く）
1. **プロセス/ポートの競合**: 異常終了した以前のプロセスが残留し、必要なポート（3972, 18789）を占有していたため、新規のサービス起動が阻害されていた。
2. **パスの不一致と固定化**:
   - `start.sh` 内の相対/絶対パス指定と実際のコード位置（`src/server.js`など）の不一致。
   - `interactive-setup.js` におけるコピー失敗時のフォールバック処理により、システム全体から見たインストール場所が本来のグローバルパスとずれている状態。
   - `~/.openclaw/adapter-node-path.txt` にNode.jsのパスが固定されていることによる、稼働環境（NVMなどの切り替え後）とのバイナリパスのズレ。
3. **認証とアクセス状態**: OpenClaw Gatewayへのアクセスには特定の認証トークンが付与されたURL（`#token=...`）が必要な仕様であるが、エラーログや接続拒否の状態は、トークン未付与の直接アクセスや、Gatewayサービス自体が正しくリッスンしていないことに起因する事象として記録されている。
