### 調査レポート：[OpenClaw Control UI におけるHTMLコメント露出問題と SSoT 3.1 ゼロ幅文字ステガノグラフィの検証]

【調査の道筋（仮説と検証のトレイル）】
- **仮説**: SSoT 3.0 で導入された HTMLコメント (`<!-- <tool_metadata>...</tool_metadata> -->`) によるメタデータインライン埋め込み手法について、ユーザーから「UI上で見えてしまっている」との報告があった。OpenClaw のフロントエンド（Markdownレンダラ）がセキュリティ上の理由からHTMLエスケープを実行している可能性がある。
- **検証**: 実際にUI上に表示されたログのスクリーンショットやユーザーの報告を確認。さらにアダプタ側（`streaming.js`）が出力している生のSSEストリームテキスト構造を検証した。
- **事実の発見**: アダプタからの送信は意図通り行われていたが、UI側のMarkdownパーサーの干渉により「画面に表示されないメタデータ」という前提が崩れている事実が判明した。同期仕様上「UIに送る文字＝次ターンの履歴コンテキストになる文字」であるため、送信内容自体をサーバー側で不可視フィルターにかける手法は実行不可能であるという制約事実を再確認した。

【追加の議論と検証ループ】
- **議論/指摘**: UI表示を防ぐためにインラインデータ送信自体をやめることはSSoT（ステートレス）の破壊につながる。UI側で絶対にサニタイズされず、かつ幅を持たない形式でのテキストエンコードが必要であるという要件。
- **新仮説**: 表示幅を持たないUnicodeのゼロ幅文字（ZWC: Zero-Width Characters）である `\u200B` (Zero Width Space) や `\u200C` (Zero Width Non-Joiner) を利用し、JSONをバイナリ変換して文字列の末尾にステガノグラフィとして付加すれば、UI上のMarkdownパーサーをすり抜け完全な不可視を達成できるのではないか。
- **再検証と事実**: 
  1. Node.js スクリプトを用いて ZWC エンコーダ・デコーダのプロトタイプを作成し、送受信変換ロジックが成立することを確認した。
  2. `src/streaming.js` にこの `encodeZwc` / `decodeZwc` 処理を実装し、コンテナを再ビルド後 `curl` によるストリーム出力（`test_zwc1.txt`）をパース検証した。
  3. その結果、テキスト出力中に膨大なZWCチャンクが含まれるものの、画面上は完全不可視となることが実証された。
  4. さらに、ZWCが含まれたテキストをそのまま過去メッセージとして渡し `test_zwc2.txt` を生成したところ、Gemini CLI側が正確にZWCをデコードし、メタデータ（`toolCalls` と `functionResponse`）としてアタッチ・復元できるという事実を確認した。

【追加の議論と検証ループ】※実環境での動作確認（ユーザー報告）
- **議論/指摘**: 実際にOpenClawのControl UIから指示を出し、`read_file` や `google_web_search` が連続実行されるマルチターンシナリオにおいて、ZWCメタデータが維持・パースされるか。
- **新仮説**: SSoT 3.1 実装が本番UI環境でもハルシネーション（記憶喪失）を引き起こすことなく完全にネイティブ履歴として吸収されるはずである。
- **再検証と事実**: コンテナ内の `.docker-config/openclaw/workspace/memory-injected` の出力事実（ログファイル本体）を `view_file` で閲覧した。
- **新たな決定的な事実**: `memory-injected` ファイルの出力内容（行4900以降）において、ZWC形式から復号されたJSON内容が、Gemini CLIネイティブの `toolCalls` オブジェクト配列として文字通り一言一句狂いなく完璧なJSON構造体としてアタッチ・保存されている事実を確認した。文中の「⚙️ Using tool...」等の装飾ノイズも完全に除去（クレンジング）された状態で保存されている事実が確認された。

【コードベース・仕様から確認された最終的な事実】
- `src/streaming.js` の6行目〜24行目にて、ZWCを用いた `encodeZwc`, `decodeZwc` ロジックが実装されている。
- 同じく `src/streaming.js` の135行目付近（`messages.map`による再構築部）にて、正規表現 `new RegExp('${ZWC_START}([\\u200B\\u200C\\u200D]+)${ZWC_END}', 'g')` によってZWCブロックを抽出し、`toolCalls` オブジェクトへ復元（デコード・アタッチ）する処理が実装されている。
- これにより、Adapterは完全にステートレスでありながら、ZWCテキストバッファのみを記憶媒体（SSoT）として用いる「ハイブリッド履歴同期基盤 (SSoT 3.1)」が成立・稼働している事実が仕様およびログ出力内容から立証された。
