# 修正報告: ZWCパースエラーによるシステムのハングアップと通信整合性問題の完全解消 (SSoT 4.0)

## 発生した問題
OpenClaw Gateway経由で「返信が返ってこない（ハングする）」、あるいは「数回会話するとエラーが出る」現象が発生しました。調査の結果、以下の4つの問題が連鎖してシステムを停止させていたことが判明しました。

1. **TCPチャンク分割による文字化け (根本原因1):**
   巨大なデータを受信する際、3バイトで構成されるZWC（ゼロ幅文字）の途中でデータが分割され、文字化けが発生。これによりJSON構造が破壊されていた。
2. **OpenClawのバッファ制限によるデータ破壊 (根本原因2):**
   ツール実行結果をZWCに詰め込みすぎたため、OpenClaw Gatewayの送信制限（約32KB）に抵触し、末尾が物理的にカットされていた。
3. **OpenClawのワードラップによる強制改行 (根本原因3):**
   スペースのない長いZWC文字列に対し、OpenClawが自動的に「改行」を挿入。これがJSON内に不正な制御文字として混入し、パースエラーを引き起こしていた。
4. **ランナープロセスの「Ready」検知失敗:**
   IPC通信の不確実性により、ランナーが起動しても親プロセスがそれを認識できず、ハングアップ状態に見えていた。

## 解決策: SSoT 4.0 (サーバーサイド・コンテキスト・リハイドレーション)
「巨大なデータをOpenClawに預けない」という方針へアーキテクチャを転換しました。

1. **完全ポインタ化 (`tool_use` / `tool_result`):**
   ツール実行の指示と結果の両方をアダプター側（`logs/contexts/`）に保存し、OpenClawへはIDのみを送信。これにより、通信データを劇的に軽量化（数百バイト）し、あらゆる制限を回避。
2. **サーバーサイド・リハイドレーション:**
   OpenClawから届いたIDを元に、アダプターが送信直前に実データをローカルから読み戻してGeminiに渡すことで、完璧な記憶を維持。
3. **ノイズ除去フィルターの導入:**
   デコード前にZWC以外の文字（強制改行やスペース）を全て一掃するロジックを追加し、OpenClawによる文字加工への絶対的な耐性を獲得。
4. **デュアル待機方式:**
   IPCメッセージに加え、標準出力の文字列検知によるReady判定を導入し、ランナーの起動を確実に保証。

## 修正済みファイル (バックアップ)
以下のファイルを `fixes/modified_files/` にコピーしました。

*   `src/server.js`: `Buffer.concat` による文字化け防止。
*   `src/streaming.js`: SSoT 4.0 (ポインタ・復元) およびノイズ除去フィルターの実装。
*   `src/runner-pool.js`: ランナー待機ロジックの堅牢化。

## 結果
最終テストの結果、パースエラー（`zwc_errors.log`）は **0バイト（皆無）** となり、巨大なファイルを扱ってもハングアップせず、かつGeminiが全ての文脈を正しく記憶していることが立証されました。
